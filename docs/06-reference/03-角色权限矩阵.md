# 角色权限矩阵

> 本文档描述 RSDP 系统的角色、权限码以及角色与权限的对应关系，是后端 `SecurityConfig`、方法级 `@PreAuthorize`、前端路由与按钮权限的统一参考。
>
> 数据权威来源：
> - 后端权限常量：`server/src/main/java/com/rsdp/security/Permissions.java`
> - 数据库角色/权限初始化：`database/seed_data.sql`
> - 数据范围计算：`server/src/main/java/com/rsdp/security/datascope/DataScopeContext.java`

---

## 一、角色说明

| 角色编码 | 角色名称 | 说明 |
|----------|----------|------|
| `ADMIN` | 系统管理员 | 拥有全部权限，可管理用户、角色、系统级配置。 |
| `EDITOR` | 编辑员 | 平台运营人员，负责产品/工厂元数据维护、复核、搭配方案管理。 |
| `VIEWER` | 浏览者 | 只读角色，可浏览产品、工厂、方案等公开数据。 |
| `FACTORY_ADMIN` | 工厂管理员 | 负责维护本厂资料、本厂 RSKU 报价以及本厂已报价产品的元数据。 |
| `DESIGNER` | 设计师 | 主要使用搭配/报价功能，可读全部公开数据并管理自己的方案与设计师画像。 |
| `USER` | 普通用户 | 默认只读角色，权限与 `VIEWER` 基本一致。 |

### 2.9 用户中心-企业团队（阶段 5）

企业团队端点（`/api/v1/member/**`）**不新增权限码**：登录即可访问（与 `designer-profiles/me`
自服务模式一致），写操作由 Service 层按「企业管理员（owner）或平台 ADMIN」做归属校验。

| 端点 | 说明 | 校验规则 |
|------|------|----------|
| `PUT /member/certified-designer` | 认证设计师一键升级 | 本人；VIEWER/USER 补 DESIGNER 角色 |
| `GET/POST/PUT/DELETE /member/company` | 企业查询/创建/更新/删除 | 读=成员；写=企业管理员或 ADMIN |
| `PUT /member/company/owner` | 变更企业管理员 | 企业管理员或 ADMIN |
| `GET/POST/PUT/DELETE /member/groups` | 企业分组管理 | 读=成员；写=企业管理员或 ADMIN |
| `GET/POST/DELETE /member/members`、`PUT /member/members/*/group` | 成员管理 | 读=成员；写=企业管理员或 ADMIN |
| `GET /member/invites` | 我的邀请记录 | 本人 |

公开注册 `POST /api/v1/auth/register` 免登录，注册即 VIEWER 角色（最小权限）。

## 四、数据范围规则

权限矩阵只回答「能不能做某类操作」；对于 `FACTORY_ADMIN`，还要叠加「能不能操作某条具体数据」的数据范围约束。

### 4.1 平台运营人员（`ADMIN` / `EDITOR`）
- 数据范围：`ALL`
- 可读写所有产品、工厂、RSKU、方案等数据。

### 4.2 工厂管理员（`FACTORY_ADMIN`）
- 数据范围：`FACTORY_LIST`（仅关联的工厂）。
- **工厂资料**：只能修改/删除本厂档案（`factory_code` 在关联列表中）。
- **RSKU**：只能新增/修改/删除本厂报价。
- **产品元数据（RSPU）**：只有当本厂已对该 RSPU 有 RSKU 报价记录时，才能修改该产品元数据、变体、搭配关系。
- **删除 RSPU**：若该产品还有其他工厂报价，则单个工厂管理员不能删除整条 RSPU。

### 4.3 设计师（`DESIGNER`）
- 数据范围：当前按 `ALL` 只读处理（可读全部公开数据）。
- 写操作仅针对自己的方案、报价单、设计师画像等自有数据。
- 后续如需限制为仅看自己创建的数据，需在相关表补充 `created_by` 字段并改为 `SELF_CREATED` 范围。

### 4.4 浏览者（`VIEWER`）/ 普通用户（`USER`）
- 数据范围：`PUBLIC_ONLY`（实际只返回有权限读取且未被软删除的公开数据）。
- 只读，不能执行任何写操作。

## 五、权限前缀约定

| 场景 | 前缀 | 示例 | 说明 |
|------|------|------|------|
| `hasRole()` | `ROLE_` | `hasRole("FACTORY_ADMIN")` 实际检查 `ROLE_FACTORY_ADMIN` | Spring Security 自动拼前缀，仅用于角色判断。 |
| `hasAuthority()` | 无 | `hasAuthority("product:read")` | 用于检查权限码。 |

> 后端 `JwtAuthenticationFilter` 会把数据库中的 `role_code` 映射为 `ROLE_xxx` 的 Authority；权限码则原样作为 Authority 写入。
>
> 因此 Controller 中优先使用 `hasAuthority(T(com.rsdp.security.Permissions).PRODUCT_READ)` 等方式引用权限常量，避免硬编码字符串。

## 六、前端权限使用指南

- 路由权限：见 `web/src/router/index.ts`，通过 `meta.permissions` / `meta.roles` 控制页面级访问。
- 按钮权限：页面内使用 `useUserStore().hasPermission()` / `hasAnyPermission()` / `hasAnyRole()` 判断。
- 数据范围相关按钮显隐：除了权限判断，还要结合「是否关联本厂」等数据范围条件（如 `ProductDetailView.vue` 中的 `isAssociated`）。
- 403 提示：API 拦截器会在收到 HTTP 403 时统一弹出提示；路由级无权限会跳转 `/403` 页面。
