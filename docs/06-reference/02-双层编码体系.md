## 一、双层编码体系（最终版）

表格



| 层级     | 定义                   | 编码规则                                  | 示例                  | 存在表        |
| :------- | :--------------------- | :---------------------------------------- | :-------------------- | :------------ |
| **RSPU** | 设计原型（同款概念）   | `[类别2位]-[风格2位]-[流水3位]-[尺寸1位]` | `FS-MC-001-M`         | `rspu_master` |
| **RSKU** | 供应单元（某厂某材质） | `[RSPU ID]-[工厂代码3位]-[材质版本2位]`   | `FS-MC-001-M-A004-PE` | `rsku_supply` |

**铁律**：

- `rspu_master` 里**绝对不存在**工厂代码、价格、交期、SKU
- `rsku_supply` 里**必须外键关联** `rspu_id`，且自身携带工厂+材质+价格

## 三、SQLite 建表 SQL（RSDP 1.0 本地版）

sql

```sql
-- ============================================
-- RSPU 设计原型主表（款式主档案）
-- 绝对不含工厂、价格、SKU 信息
-- ============================================
CREATE TABLE rspu_master (
    -- 基础身份层（Identity）
    rspu_id TEXT PRIMARY KEY,              -- FS-MC-001-M
    category_path TEXT NOT NULL,           -- JSON: ["家具","座椅","休闲椅","单椅"]
    category_vectors TEXT,                 -- JSON: {"l1":0.1,"l2":0.3,"l3":0.9}
    tier TEXT,                             -- S/A/B/C（设计原型默认等级，通常取最高供应等级）
    recommendation_pool TEXT,              -- JSON: ["中古风核心","客厅单椅"]
    status TEXT DEFAULT 'active',          -- active/discontinued/draft
    
    -- 美学属性层（Aesthetics）
    style_vector TEXT,                     -- JSON: [512个浮点数] ← Agent自动生成
    style_label_human TEXT NOT NULL,       -- 9大风格标签
    six_dim_tags TEXT,                     -- 人类可读：轮廓/靠背/扶手/腿部/材质/软包
    color_primary_name TEXT,               -- 主色名：焦糖棕
    color_primary_hsv TEXT,                -- JSON: [H,S,V]
    color_secondary TEXT,                  -- JSON: ["米白","原木"]
    material_tags TEXT,                    -- JSON: ["柚木","藤编"]（设计原型典型材质）
    scene_tags TEXT,                       -- JSON: ["客厅","书房","咖啡厅"]
    white_bg_image_path TEXT,              -- 参考图路径
    aesthetics_confidence TEXT,            -- high/mid/low
    
    -- 空间物理层（Spatial）
    product_dimensions TEXT,               -- JSON: {"w":560,"d":580,"h":780,"unit":"mm"}
    occupancy_static TEXT,                 -- JSON
    occupancy_dynamic TEXT,                -- JSON（含拉出空间）
    packaging TEXT,                        -- JSON（典型包装尺寸）
    hardscape_constraints TEXT,            -- JSON
    spatial_relationships TEXT,            -- JSON
    
    -- 业务规则层（Business Logic）
    reference_price_band TEXT,             -- low/mid/high（基于历史报价区间）
    tier_badge TEXT,                       -- A级质价比
    warranty_years INTEGER,                -- 典型质保年限
    key_specs TEXT,                        -- JSON: {"frame":"白蜡木","foam_density":"45D"}
    style_compatibility TEXT,              -- JSON: ["中古风","侘寂风"]
    budget_range TEXT,                     -- JSON: [800,2500]
    
    -- 元数据
    created_date DATE DEFAULT CURRENT_DATE,
    updated_date DATE,
    source_agent_version TEXT              -- 如 "qwen3.5-vl-7b-v1.0"
);

-- RSPU 索引
CREATE INDEX idx_rspu_style ON rspu_master(style_label_human, six_dim_tags);
CREATE INDEX idx_rspu_color ON rspu_master(color_primary_hsv);
CREATE INDEX idx_rspu_status ON rspu_master(status);

-- ============================================
-- RSKU 供应单元子表（工厂报价子表）
-- 每行 = 一个工厂对一个RSPU的一种材质报价
-- ============================================
CREATE TABLE rsku_supply (
    -- 供应身份
    rsku_id TEXT PRIMARY KEY,              -- FS-MC-001-M-A004-PE
    rspu_id TEXT NOT NULL,                 -- 外键，关联RSPU
    factory_code TEXT NOT NULL,            -- S001/A004/B007/C012
    material_code TEXT NOT NULL,           -- PE/LE/TN/LI 等
    
    -- 供应链层（Supply Chain）
    factory_sku TEXT,                      -- 工厂原始编码 bf001
    factory_price REAL,                    -- 出厂价（明文，本地绝密）
    price_band TEXT,                       -- low/mid/high（基于出厂价自动分档）
    material_description TEXT,             -- 工厂提供的详细材质说明
    lead_time_days INTEGER,                -- 交期
    moq INTEGER,                           -- 起订量
    
    -- 品质与合规
    structure_strength_rating TEXT,        -- 家用结构/商用结构/需验证
    flame_retardant_capability TEXT,         -- 可做有案例/可做无案例/不可做
    factory_photo_path TEXT,               -- 该厂实拍图路径
    
    -- 动态评分（Performance）— 1.0手动录入，2.0自动
    factory_credit_score INTEGER,          -- 履约评分
    on_time_rate REAL,                     -- 准时率
    quality_return_rate REAL,              -- 退货率
    
    -- 元数据
    price_updated DATE,                    -- 价格更新日期
    quote_confidence TEXT,                 -- high/mid/low
   复核状态 TEXT DEFAULT '待复核',         -- 待复核/已确认/存疑
    created_date DATE DEFAULT CURRENT_DATE,
    
    FOREIGN KEY (rspu_id) REFERENCES rspu_master(rspu_id)
);

-- RSKU 索引
CREATE INDEX idx_rsku_rspu ON rsku_supply(rspu_id);
CREATE INDEX idx_rsku_factory ON rsku_supply(factory_code);
CREATE INDEX idx_rsku_price ON rsku_supply(price_band, factory_price);
CREATE INDEX idx_rsku_material ON rsku_supply(material_code);

-- ============================================
-- 工厂档案表（独立维护，不依附RSPU/RSKU）
-- ============================================
CREATE TABLE factory_master (
    factory_code TEXT PRIMARY KEY,         -- S001/A004
    factory_name TEXT,
    factory_level TEXT,                    -- S/A/B/C
    home_commercial_tag TEXT,              -- 家用级/商用级
    certification TEXT,                    -- JSON: ["BSCI","ISO9001"]
    engineering_cases TEXT,              -- 工程案例清单
    region TEXT,                           -- 佛山龙江/东莞厚街
    contact_person TEXT,
    first_audit_date DATE,
    next_visit_date DATE,
    status TEXT DEFAULT 'active'
);
```

## 六、Agent 自动生成 ID 逻辑（Prompt 可用）

plain

```plain
【RSPU ID 生成规则】
1. 判定产品类别 → FS=座椅
2. 判定风格 → MC=中古风
3. 查 SQLite：SELECT MAX(rspu_id) FROM rspu_master WHERE rspu_id LIKE 'FS-MC-%'
   → 返回 FS-MC-005-M，新流水号 = 006
4. 判定尺寸档：
   - 餐椅/书椅/无扶手矮椅 → S
   - 标准休闲椅/扶手椅 → M
   - 宽体休闲椅/大座深 → L
   - 躺椅级别宽大单椅 → X
5. 生成 RSPU ID：FS-MC-006-M
6. 写入 rspu_master

【RSKU ID 生成规则】
1. 继承 RSPU ID：FS-MC-006-M
2. 工厂代码：员工提供 A004 → 直接拼接
3. 材质版本码：Agent 根据材质描述判定
   - "PE仿藤" → PE
   - "油蜡皮" → LE
   - "真藤" → TN
   - "羊羔绒" → SF
4. 生成 RSKU ID：FS-MC-006-M-A004-PE
5. 写入 rsku_supply，外键 rspu_id = FS-MC-006-M

【同款判定规则】
新品图 vs 现有 RSPU 库：
- 先 SQLite 预筛：同风格 + 同轮廓 + 同靠背 + 同扶手
- 再 Agent 多图终审：综合相似度 ≥ 85% → 沿用现有 RSPU ID，只生成新 RSKU ID
- 综合相似度 < 85% → 生成新 RSPU ID + 新 RSKU ID
```

家具场景六维标签与风格判断机制



Kimi 这类多模态 Agent 拿到一张产品图后，内部经历了 **"像素 → 视觉token → 语义概念 → 结构化标签"** 四层转换。它不是用尺子量，也不是查数据库比对，而是靠预训练时"看过"的数百万张家具图形成的**模式直觉**。

## 一、底层机制：Agent 怎么"看"图

plain

```plain
产品白底图（jpg/png）
    │
    ▼
【视觉编码器】把图片切成 16×16 的小方块，每个方块转成数字向量
    │
    ▼
【跨模态对齐】图片向量与文字向量映射到同一个语义空间
    │
    ▼
【大语言模型】基于预训练知识，把图片向量"翻译"成人类语言
    │
    ▼
输出："这是一把昌迪加尔椅，V型支撑腿，藤编靠背，柚木框架..."
```

**关键**：Agent 从未见过你工厂这张具体的图，但它见过**昌迪加尔椅的上万张网图、设计稿、博物馆照片**，所以能靠"像素模式的统计相似性"认出来。

## 二、特征维度判断：从像素到六维标签

当你要求 Agent 按"轮廓/靠背/扶手/腿部/材质/软包"六维输出时，它内部做了**语义拆解**：

表格



| 六维        | Agent 内部推理链                                    | 示例                                              |
| :---------- | :-------------------------------------------------- | :------------------------------------------------ |
| **A. 轮廓** | 先看整体剪影 → 是几何块面？有机曲线？还是对称包裹？ | "整体呈A字架展开，上宽下窄" → **A字架形**         |
| **B. 靠背** | 看背部高度与结构 → 高过头顶？齐肩？镂空？软包？     | "靠背由藤编网格构成，无软垫" → **编织镂空**       |
| **C. 扶手** | 看手臂支撑部位 → 有无？与靠背是否连体？厚度？       | "扶手与靠背一体，向外微扩，平板状" → **平板直扶** |
| **D. 腿部** | 看支撑结构 → 几条腿？角度？材质？有无底座？         | "两腿呈八字外扩，与扶手连为一体" → **八字外扩**   |
| **E. 材质** | 看表面反光与纹理 → 木纹？编织纹？金属光泽？塑料感？ | "哑光木纹+棕黄色编织肌理" → **藤编+实木纹理**     |
| **F. 软包** | 看座面/靠背是否有填充物 → 硬座？薄垫？厚包裹？      | "座面为硬板，无海绵填充" → **无软包硬座**         |

**这不是精确测量**，而是**视觉语义的"最像哪个词"**。Agent 在预训练时学过"藤编=交织纹理+哑光+棕黄"、"钢管=高反光+直线+银灰"，所以看到类似像素分布就触发对应词汇。

## 三、风格分类：9选1是怎么决定的

Agent 做风格分类时，不是"算分排序"，而是**多特征加权投票**：

plain

```plain
看到图片特征：
├─ 柚木 + 藤编 + V型腿 + 1950年代感 → 中古风 +3票
├─ 钢管 + 皮革 sling + 悬臂 → 包豪斯 +3票  
├─ 羊羔绒 + 圆润 + 奶油色 → 奶油风 +3票
├─ 纳帕皮 + 厚重 + 金属细脚 → 意式 +3票
└─ ...
```

**决策逻辑**：

- 某风格得票 **显著高于** 其他（如中古 8 票 vs 侘寂 2 票）→ **高置信度**，直接输出
- 两风格票数接近（中古 5 票 vs 侘寂 4 票）→ **中置信度**，输出"中古风（疑似侘寂边缘）"
- 特征混杂无法归类（如"钢管+藤编+油蜡皮+羊羔绒"混搭）→ **低置信度**，输出"需人工确认"

## 四、同款比对：85%相似度怎么算

Agent 做"是否同款"时，内部没有精确算法，而是**语义描述的重合度**：

plain

```plain
现有款 MC-001 的六维标签：A字架形/编织镂空/平板直扶/八字外扩/藤编+实木/无软包
新品图的六维标签：        A字架形/编织镂空/平板直扶/八字外扩/PE仿藤+实木/无软包

比对：
├─ A轮廓：相同 → +20%
├─ B靠背：相同 → +20%
├─ C扶手：相同 → +20%
├─ D腿部：相同 → +15%
├─ E材质：藤编 vs PE仿藤（同类不同质）→ +10%
└─ F软包：相同 → +15%

综合：100%？不，Agent会降权材质差异 → 输出"约90%，建议关联现有款"
```

**注意**：Agent 的"相似度"是**语义近似估算**，不是 3060 跑 CLIP 向量的**数学精确距离**。所以：

- Agent 说 90% → 大概率真的是同款
- Agent 说 70% → 可能是同款不同材质，也可能是近似款，**必须人工看图确认**

表格



| 场景                                        | Agent 表现                       | 原因                     |

| **D. 腿部/底座** | **四直腿**、**圆盘底座**、**方盘底座**、**悬臂单柱**、**X型交叉**、**八字外扩** | 底座类型是吧椅核心识别点 |
| **E. 表面材质**  | 实木纹理、金属镜面/做旧、真皮光泽、超纤皮、绒布、藤编/仿藤、亚克力 | 看座面及靠背材质         |
| **F. 软包形态**  | 无软包硬座、薄垫平板、一体定型棉、分体坐垫、全包裹           | 看座面是否有填充         |

**关键区别**：吧椅的 **D 维度（腿部/底座）** 比休闲椅更重要，因为底座类型直接决定商用稳定性（圆盘底座适合旋转，四直腿适合堆叠）。

## 三、吧椅 RSKU 编码（工厂+材质+功能）

吧椅的 RSKU 在标准 RSDP 基础上，**增加功能码段**：

plain

```plain
BS-IN-001-7-A004-LE-R
```

表格



| 增量段              | 含义         | 选项                                                         |
| :------------------ | :----------- | :----------------------------------------------------------- |
| **第5段：工厂代码** | 同前         | `S001`/`A004`/`B007`                                         |
| **第6段：材质码**   | 吧椅专用     | `LE`=皮革 / `WO`=实木 / `MT`=金属 / `VN`=天鹅绒 / `MS`=仿皮 / `RB`=藤编 |
| **第7段：功能码**   | **吧椅特有** | `R`=旋转 / `L`=升降 / `F`=固定 / `S`=可堆叠 / `C`=带脚踏环   |

**示例对比**：

表格



| RSKU ID                 | 完整语义                                            |
| :---------------------- | :-------------------------------------------------- |
| `BS-IN-001-7-A004-LE-R` | 工业风吧椅001-75cm-A004厂-皮革-旋转款               |
| `BS-IN-001-7-A004-LE-F` | 同款同厂同材质，但**固定底座**（不旋转，便宜200元） |
| `BS-IN-001-7-B007-MT-S` | B007厂金属版-可堆叠（适合连锁餐饮批量收纳）         |

## 四、同空间吧椅的分类策略（以酒吧为例）

同一个酒吧空间里，吧椅可能有 3-5 种不同"角色"，分类时要**按功能位区分**，而不是按风格区分（风格通常统一）：

表格



| 空间位置                   | 吧椅角色               | 分类维度           | RSPU 示例                         |
| :------------------------- | :--------------------- | :----------------- | :-------------------------------- |
| **吧台主位**（调酒师对面） | 高频坐、需旋转、需舒适 | 旋转+软包+标准高   | `BS-IN-002-7`（旋转皮革中背）     |
| **吧台侧位**（靠墙长条）   | 中频坐、固定、省空间   | 固定+无扶手+窄座   | `BS-IN-003-7`（固定平板无扶手）   |
| **高脚桌散台**（窗边）     | 低频坐、可堆叠、易清洁 | 可堆叠+塑料/金属   | `BS-MN-001-7`（伊姆斯塑料吧椅）   |
| **外摆区**（露台）         | 防水防晒、耐候         | PE藤/不锈钢/户外布 | `BS-WJ-001-6`（矮背仿藤户外吧椅） |
| **VIP包厢吧台**            | 厚重感、身份暗示       | 实木+皮革+高背包裹 | `BS-MC-001-8`（高背实木皮吧椅）   |

**分类原则**：

1. **先按座高锁定**（酒吧吧台多高，吧椅必须匹配，误差±3cm）
2. **再按空间位置定功能**（旋转/固定/堆叠）
3. **最后按风格统一**（一个酒吧通常只选1-2种风格，避免杂乱）

吧椅也有风格，但**风格服务于空间调性**，不像家装那样自由混搭：

表格



| 商业空间类型   | 主流吧椅风格 | RSPU 风格码 | 典型材质组合            |

## 一、产品向量数据库（Product Vector Database）

### 1.1 它是什么？

不是传统表格里的"行和列"，而是把每款产品变成**数学空间里的一个点**。这个点有 512 个坐标（维度），每个坐标代表一个视觉/语义特征。

表格



| 传统数据库                    | 向量数据库                                        |
| :---------------------------- | :------------------------------------------------ |
| 查"中古风" → 精确匹配文字标签 | 查"中古风" → 找与"中古风锚点"**距离最近**的所有点 |
| 同款判定靠人工看图            | 同款判定靠算**余弦相似度**                        |
| 只能搜已知标签                | 能搜**近义、跨界、混搭**（如"中古+侘寂混血"）     |

### 1.2 512 维里到底存了什么？

CLIP 模型（或 BGE-VL）把产品图编码后，512 个数字大致对应这些**语义方向**（人类可理解的近似解释）：

表格



| 维度区间 | 代表特征  | 吧椅/休闲椅示例                    |
| :------- | :-------- | :--------------------------------- |
| 0-63     | 整体轮廓  | 蛋形 vs 方盒 vs A字架              |
| 64-127   | 材质纹理  | 藤编交织度、木纹粗糙度、金属反光率 |
| 128-191  | 色彩分布  | 主色 HSV、渐变、饱和度             |
| 192-255  | 结构比例  | 座高/背高比、腿粗度、扶手外扩角    |
| 256-319  | 风格语义  | 复古权重、工业权重、极简权重       |
| 320-383  | 工艺细节  | 榫卯 vs 焊接 vs 注塑、做旧程度     |
| 384-447  | 空间尺度  | 小巧感 vs 厚重感、通透 vs 封闭     |
| 448-511  | 情绪/调性 | 温暖、冷峻、戏谑、沉稳             |

**关键**：这些维度不是人工标注的，是模型在预训练时**自动涌现**的。人类只能近似解读，但机器靠它们做精确计算。

### 1.3 在 Windows 上的实现（ChromaDB）

plain

```plain
Windows 工作站（推荐 NVIDIA RTX 3060/4060 或更高，16G+ 内存）
├── ChromaDB（向量数据库，Docker 或本地服务）
│   └── collection: "furniture_vectors"
│       ├── id: "FS-MC-001-M" (RSPU ID)
│       ├── embedding: [0.12, -0.05, 0.88, ...] (512维)
│       ├── metadata: {"style":"中古风", "category":"休闲椅", "color_hsv":"[30,0.6,0.4]"}
│       └── document: "昌迪加尔椅，柚木真藤，V型腿..." (可检索文本)
│
└── 查询时
    ├── 输入：新品图 → SpringBoot 调用 Ollama /api/embeddings → 512维向量
    ├── 操作：collection.query(embedding=new_vector, n_results=5)
    └── 输出：Top 5 最相似的现有 RSPU + 相似度分数
```

**查询速度**：10 万张产品图，单次比对 **< 100 毫秒**。

### 1.4 在你的工作流里的三个应用

表格



| 应用         | 向量操作                                                    | 业务价值                         |
| :----------- | :---------------------------------------------------------- | :------------------------------- |
| **同款判定** | 新品向量 vs 全库向量，找距离 < 0.15 的                      | 防止重复建档，同款不同厂自动关联 |
| **风格迁移** | 客户现有沙发向量 + 偏移量 → 推荐"同风格不同功能"产品        | 自动配休闲椅/茶几/边几           |
| **反查竞品** | 客户发来一张 Pinterest 网图 → 编码 → 搜库找最接近的自家产品 | 快速响应"我要这个感觉的"         |

## 二、空间约束（产品与环境的空间耦合）

这是 RSDP `spatial` 层中最容易被忽略、但**落地时最容易翻车**的部分。它回答：**这个产品放进具体空间后，会与硬装/环境发生什么物理关系？**

### 2.1 与地面耦合（Floor Coupling）

表格



| 约束类型         | 数据字段             | 你的项目示例                                                 |
| :--------------- | :------------------- | :----------------------------------------------------------- |
| **承重**         | `floor_load` (kg/m²) | 实木大班台 80kg 集中在 0.5m² → 局部承重 160kg/m²，老楼板需预警 |
| **地面材质适配** | `floor_compatible`   | 酒吧水泥自流平地面 → 金属细腿椅需配毛毡垫，防刮花            |
| **找平要求**     | `level_tolerance`    | 四条腿长度微调范围（如 ±5mm），应对不平地面                  |
| **固定需求**     | `anchor_required`    | 高吧椅圆盘底座在光滑地面是否需防滑垫/固定螺丝                |

### 2.2 与墙面耦合（Wall Coupling）

表格



| 约束类型                 | 数据字段                                  | 你的项目示例                                                 |
| :----------------------- | :---------------------------------------- | :----------------------------------------------------------- |
| **靠背高度 vs 插座高度** | `backrest_height`, `wall_socket_conflict` | 休闲椅高背 90cm，墙面插座在 75cm → `conflicts_with: ["强电插座"]` |
| **靠墙摆放**             | `wall_clearance`                          | 蛋椅旋转底座需离墙 ≥ 15cm，否则旋转撞墙                      |
| **护墙板/踢脚线**        | `skirting_compatibility`                  | 法式护墙板凸出 3cm，椅子靠背需内凹或加垫防磕碰               |
| **窗边紫外线**           | `uv_exposure_tolerance`                   | 靠窗摆放的真皮椅，日晒 3 年褪色 → 材质标签需预警             |

### 2.3 与顶面耦合（Ceiling Coupling）

表格



| 约束类型       | 数据字段                  | 你的项目示例                                                 |
| :------------- | :------------------------ | :----------------------------------------------------------- |
| **吊灯垂高**   | `ceiling_light_clearance` | 高背休闲椅上方有吊灯，椅背最高点 + 吊灯最低点 ≥ 40cm         |
| **层高压迫感** | `visual_height_ratio`     | 吧椅座高 75cm + 靠背 45cm = 总高 120cm，层高 2.4m 时视觉比例 50%，偏压抑 |

### 2.4 水电点位耦合（MEP Coupling）

这是商业空间（酒吧/会所）的**致命细节**：

plain

```plain
产品：吧台操作台
├── power_requirement: {"voltage":220, "watts":2000, "outlets":4, "position":"背板下方30cm"}
├── plumbing_requirement: {"drain":true, "inlet":"DN15", "position":"距左端50cm"}
└── network_requirement: {"ethernet":2, "position":"背板内侧"}

环境约束：
├── 现场地面已有排水沟 → 吧台排水管需对接，距离>2m则无法安装
├── 现场墙面插座高度60cm → 吧台设备插座需求高度30cm → 需要改电位或拖线板
└── 现场无网络口 → 收银系统无法联网 → 产品选型需换无线版本或增加AP
```

**RSDP 中的表达**：

JSON

```json
"hardscape_constraints": {
  "floor_load": 50,
  "wall_mount": false,
  "ceiling_mount": false,
  "power_requirement": {"voltage":220, "amps":10, "position":"背板下方30cm"},
  "plumbing_requirement": {"drain":true, "inlet":"DN15"}
}
```

## 三、物理占位（产品与其他产品的配合）

这是**动态空间计算**的核心。RSDP 把产品从"孤立 SKU"变成"空间占位符"，占位分两层：

### 3.1 静态占位（Static Occupancy）

产品本身占用的**最小矩形空间**：

plain

```plain
昌迪加尔椅：
├── product_dimensions: {"w":560, "d":580, "h":780}
└── static_occupancy: {"w":560, "d":580, "h":780}
```

**用途**：计算"这个房间能摆几把"。

### 3.2 动态占位（Dynamic Occupancy）

产品**使用时实际占用的空间**，包含操作半径：

表格



| 产品       | 静态占位 | 动态占位          | 计算逻辑                         |
| :--------- | :------- | :---------------- | :------------------------------- |
| 休闲椅     | 60×60cm  | 60×**90**cm       | 座深 + 腿部前伸 30cm             |
| 吧椅       | 40×40cm  | 40×**70**cm       | 座高 75cm，腿自然下垂需前方 30cm |
| 大班台     | 200×90cm | **230**×**120**cm | 椅子拉出 + 老板起身绕桌          |
| 文件柜抽屉 | 40×45cm  | 40×**90**cm       | 抽屉拉出深度 + 人蹲取文件        |
| 旋转休闲椅 | 60×60cm  | **直径90cm圆**    | 旋转扫过的完整圆                 |

**RSDP 中的表达**：

JSON

```json
"occupancy_space": {
  "static": {"w":560, "d":580, "h":780},
  "dynamic": {"w":560, "d":900, "h":780}
}
```

### 3.3 产品间配合（Spatial Relationships）

这是**组合销售/方案设计**时的自动匹配引擎：

#### A. 正向配合（Compatible With）

plain

```plain
沙发 RSPU: FS-MC-001-M (宽220×深95cm)
├── compatible_with: [
│   {"category":"茶几", "size_range":"800-1200mm", "height":"≤45cm"},
│   {"category":"边几", "size_range":"400-500mm", "position":"扶手侧"},
│   {"category":"地毯", "size_range":"2000×3000mm", "clearance":"≥10cm"}
│ ]
```

**自动计算**：Agent 看到客户选了这款沙发，自动推荐 900mm 宽、40cm 高的中古茶几。

#### B. 冲突检测（Conflicts With）

plain

```plain
高背休闲椅: FS-MC-005-L
├── conflicts_with: [
│   {"object":"暖气片", "reason":"靠背遮挡散热，距离需≥20cm"},
│   {"object":"落地窗矮台", "reason":"座高+背高>窗台高，视觉压抑"},
│   {"object":"强电插座", "reason":"靠背遮挡，使用不便"}
│ ]
```

**商业空间示例（酒吧）**：

plain

```plain
吧台吧椅 BS-IN-001-7
├── conflicts_with: [
│   {"object":"吧台下方酒柜门", "reason":"吧椅圆盘底座与柜门开启半径冲突"},
│   {"object":"相邻吧椅", "reason":"肩宽45cm+活动空间，椅间距需≥60cm"}
│ ]
```

#### C. 通道预留（Clearance）

plain

```plain
产品组合：餐桌 + 餐椅
├── clearance_front: 600mm (椅前留足拉开展位+人走过)
├── clearance_sides: 100mm (椅与墙/柜的缝隙)
└── clearance_behind: 450mm (椅后空间，仅后推时可用)
```

**商业空间（会议室）**：

plain

```plain
会议桌 4000×1400mm + 会议椅 12 把
├── 单侧动态占位：椅宽 60cm + 拉出 30cm + 人站起 45cm = 135cm
├── 房间最小净宽：桌宽 140cm + 两侧 135cm × 2 = 410cm
└── 若房间净宽仅 380cm → Agent 预警：需换窄桌或减少椅数
```

## 四、三个概念的联动应用（Agent 场景）

### 场景：客户说"我要一个 20㎡ 的威士忌吧角落"

**Agent 的推理链**：

plain

```plain
1. 产品向量数据库
```

## 一、512维的本质：不可解释的黑箱

表格



| 你的理解（合理但不对）                  | 技术真相                                             |
| :-------------------------------------- | :--------------------------------------------------- |
| "0-63是轮廓，我可以改成0-100放沙发座深" | ❌ 每个维度的含义是模型**自动涌现**的，人类无法指定   |

**CLIP/BGE 的真实工作方式**：

- 模型看过互联网上的 4 亿张图（含家具、动物、风景、人脸）
- 它自己"发现"了 512 个统计规律（比如"第 47 维似乎与圆形相关"）
- 这些规律**不是人类标注的**，也无法被人类精确控制
- 同一套 512 维，既能编码"昌迪加尔椅"，也能编码"螃蟹沙发"，只是**数值分布不同**

## 三、用户能做什么？（替代"修订维度"）

虽然你不能修订 512 维的内部结构，但你可以做**三件事**来让系统更懂你的品类：

### 1. 提供锚点图片（Anchor Images）——校准风格边界

plain

```plain
你给我 10 张"中古风沙发"的代表图
    ↓
Agent 编码 → 得到 10 个 512 维向量
    ↓
计算这 10 个向量的"中心点"（质心）
    ↓
这个中心点 = "中古风沙发"的语义锚点
    ↓
以后任何新品图，与这个锚点的距离 < 0.2 → 自动标记"疑似中古风沙发"
```

**价值**：你不需要懂 512 维，只需要提供**"这就是中古风"的样本**，系统就能学会在这个 512 维空间里画一个圈。

### 2. 定义品类专属的"元数据权重"——检索时调整优先级

虽然不能改 512 维，但可以在**检索结果排序**时，对不同品类施加不同的业务权重：

表格



| 品类       | 检索时更看重什么               | 权重调整                                    |
| :--------- | :----------------------------- | :------------------------------------------ |
| **休闲椅** | 靠背高度、扶手形态、腿部结构   | 六维标签中 B/C/D 权重 ×1.5                  |
| **沙发**   | 座深、总宽、扶手宽度、靠背高度 | 六维标签中 A（轮廓）权重 ×1.5，增加座深字段 |
| **吧椅**   | 座高、底座类型、旋转功能       | 座高档 + 功能码硬筛选                       |
| **茶几**   | 台面尺寸、高度、与沙发配合比例 | 空间约束中 clearance 权重 ×2                |

**实现方式**：不是改向量，而是改**排序算法**。

### 3. 定义品类专属的"六维标签扩展"——人类可解释层

512 维是黑箱，但 `six_dim_tags` 是**人类完全可控**的。你可以：

plain

```plain
休闲椅六维：轮廓/靠背/扶手/腿部/材质/软包
沙发六维：轮廓/靠背/座深/扶手/腿部/材质/软包/模块性  ← 加2维
茶几六维：轮廓/台面/支撑/高度/材质/收纳/配合沙发比例  ← 完全不同
```

**注意**：六维标签是**RSDP 元数据**，不是 512 维向量本身。它们用于：

- 预筛（SQL `WHERE` 条件，缩小比对范围）
- 人类可读（解释为什么两款像）
- 与 512 维向量**互补**（向量管"像不像"，六维管"为什么像"）

三层检索元数据预筛原理与实现



**元数据就是 SQLite 表格里那些人类能看懂的字段**（风格标签、六维标签、品类、价格区间、适用空间）。**元数据预筛 = 用 SQL 硬条件先把 10 万张粗暴过滤到 50 张**，再让向量库在这 50 张里精排。

## 一、元数据 vs 向量：两种数据的分工

表格



| 类型          | 存在哪里                | 长什么样                              | 谁生成         | 用途                                   |
| :------------ | :---------------------- | :------------------------------------ | :------------- | :------------------------------------- |
| **元数据**    | SQLite `rspu_master` 表 | `"中古风"`、`"A字架形"`、`[800,2500]` | Agent/员工录入 | **人类可读、SQL 可筛选、业务规则判断** |
| **512维向量** | ChromaDB 向量库         | `[0.12, -0.05, ...]`（512个浮点）     | CLIP/Qwen 编码 | **机器指纹、像素级相似度计算**         |

**关键区别**：元数据是"文字标签"，向量是"数学指纹"。预筛用标签，精排用指纹。

## 二、为什么必须先做元数据预筛？

假设你直接拿一张新品图丢进向量库搜 10 万张：

plain

```plain
向量库返回 Top 5：
1. FS-MC-001-M  昌迪加尔椅（中古休闲椅） 距离 0.08 ✅
2. SF-MC-005-L  中古皮沙发              距离 0.12 ❌（跨品类！）
3. BS-MC-002-7  中古吧椅                距离 0.15 ❌（跨品类！）
4. FS-MC-008-M  狩猎椅（中古休闲椅）    距离 0.18 ✅
5. FT-MC-003-M  中古边几                距离 0.22 ❌（跨品类！）
```

**问题**：向量只看"像素像不像"，不管这是椅子还是沙发。中古沙发和中古休闲椅在材质（皮革+实木）上像素相似，但品类完全不同。

**元数据预筛的作用**：先锁死 `品类=休闲椅` + `风格=中古风`，把 10 万张压到只有 200 张中古休闲椅，再在这 200 张里做向量比对，**彻底杜绝跨品类误匹配**。

## 三、元数据预筛的 SQL 实现（Layer 1）

这就是之前代码里 `MetadataFilter.prefilter()` 方法的本质：

sql

```sql
-- 场景：录入一把新品休闲椅，人工初步判断"中古风、A字架形、编织镂空靠背"
SELECT rspu_id, style_label_human, six_dim_tags, white_bg_image_path
FROM rspu_master
WHERE 
    rspu_id LIKE 'FS-%'                    -- 硬条件1：品类=休闲椅（FS前缀）
    AND style_label_human = '中古风'       -- 硬条件2：风格=中古风
    AND six_dim_tags LIKE '%"A":"A字架形"%'   -- 硬条件3：轮廓=A字架形
    AND six_dim_tags LIKE '%"B":"编织镂空"%'  -- 硬条件4：靠背=编织镂空
LIMIT 50;
```

**执行结果**：

- 入库前：全库 10 万张
- 执行后：只剩 **8-15 条** 同时满足"中古风+A字架形+编织镂空"的休闲椅
- 耗时：**< 10 毫秒**（有索引）

### 索引设计（预筛快的关键）

sql

```sql
-- 复合索引：品类+风格+六维标签
CREATE INDEX idx_meta_prefilter 
ON rspu_master(rspu_id, style_label_human, six_dim_tags);

-- 价格索引（如果预筛加价格条件）
CREATE INDEX idx_meta_price 
ON rsku_supply(rspu_id, factory_price);
```

## 四、三层漏斗的数据量变化

plain

```plain
全库 100,000 张产品图
    │
    ▼ 元数据预筛（SQL WHERE）
    ├─ 条件：品类=休闲椅 → 剩 15,000 张
    ├─ 条件：风格=中古风 → 剩 2,000 张  
    ├─ 条件：轮廓=A字架形 → 剩 200 张
    └─ 条件：靠背=编织镂空 → 剩 12 张
    │
    ▼ 向量精排（ChromaDB，只在12张里搜）
    ├─ 计算新品图 vs 12张候选的512维距离
    └─ 返回 Top 5（距离 0.08, 0.15, 0.22...）
    │
    ▼ LLM 终审（Qwen 7B，只比对Top 5）
    ├─ 加载新品图 + 5张参考图
    └─ 输出 JSON 判定结果
```

**漏斗效应**：

表格



| 层级       | 数据量    | 耗时        | 作用                           |
| :--------- | :-------- | :---------- | :----------------------------- |
| 元数据预筛 | 10万 → 12 | **< 10ms**  | **排错**：排除跨品类、跨风格   |
| 向量精排   | 12 → 5    | **30–80ms** | **排近**：按像素相似度排序     |
| LLM 终审   | 5 → 1     | **5–10s**   | **排细**：语义级终审，输出理由 |

**没有预筛的灾难**：向量库要在 10 万张里盲搜，虽然也能出结果，但 Top 10 里可能混进沙发/茶几/吧椅，**后期 LLM 终审的输入质量极差，误判率飙升**。

## 六、元数据预筛与向量精排的互补关系

表格



| 维度         | 元数据预筛               | 向量精排               |
| :----------- | :----------------------- | :--------------------- |
| **判断依据** | 文字标签精确匹配         | 像素模式模糊相似       |
| **速度**     | 毫秒级                   | 毫秒级                 |

| **漏报率**   | 中（六维标签写错就漏）   | 低（像素相似就能召回） |
| **最佳实践** | **先用元数据缩小范围**   | **再在范围内向量排序** |

**举例**：工厂发来一款"昌迪加尔椅变体"（腿加粗了 2cm，藤编改 PE 仿藤）。

- LLM 终审：看结构完全一致，材质只是替代 → **判定同款，建议关联**

## 二、搭配推荐的正确三层流程

plain

```plain
客户需求："中古风客厅，已有昌迪加尔椅，配茶几"
    │
    ▼
Layer 1: 元数据预筛（放宽条件）
    ├─ 条件1：风格 = 中古风（硬锁定）
    ├─ 条件2：品类 = 茶几（硬锁定）
    ├─ 条件3：scene_tags 含 "客厅"（硬锁定）
    └─ ❌ 不锁六维标签（茶几和椅子的六维完全不同）
    │
    ▼
Layer 2: 向量精排（找"和谐"而非"相同"）
    ├─ 输入：昌迪加尔椅的 512维风格向量
    ├─ 计算：与所有中古茶几的风格向量距离
    └─ 输出：Top 15 最"风格和谐"的茶几（不是同款，是搭调）
    │
    ▼
业务规则筛选（人工/脚本）
    ├─ 价格梯度过滤：预算 ±30%
    ├─ 尺寸配合过滤：茶几宽度 = 沙发宽度的 0.4-0.6
    ├─ 材质呼应过滤：已有藤编椅 → 优先藤面/实木茶几，而非玻璃/金属
    └─ 输出：3-5 款候选
    │
    ▼
资料制作 Agent（Windows 工作站）
    ├─ 生成 PPT 页：每款1页，含图+价格+材质+推荐理由
    └─ 输出：客户选型手册
```

**终审在哪里被砍掉了？** 向量精排后直接进业务规则筛选，不需要 Qwen 7B 来判定"这是不是同款"——因为**它们本来就不是同款，是不同品类的搭配**。

## 二、正向配合：审美组合如何选出？（3.3A）

不是靠拍脑袋，而是**风格向量 + 色彩向量 + 材质呼应**的三重计算。

### 算法：跨品类搭配推荐

示例代码

```text
def recommend_companion(existing_rspu_id, target_category, top_k=3):
    """
    已有产品 → 推荐搭配品类
    例：已有昌迪加尔椅 → 推荐茶几
    """
    
    # 1. 读取已有产品的 RSDP 数据
    existing = db.get(existing_rspu_id)
    existing_vector = existing.style_vector        # 512维
    existing_hsv = existing.color_primary_hsv      # [H,S,V]
    existing_material = existing.material_tags       # ["柚木", "真藤"]
    
    # 2. 元数据预筛：同风格 + 目标品类 + 同空间
    candidates = db.prefilter(
        style_label=existing.style_label,           # 中古风
        category_prefix=target_category,            # TB 茶几
        scene_tags_overlap=existing.scene_tags      # 客厅
    )
    
    # 3. 三重评分
    scored = []
    for cand in candidates:
        score = 0
        
        # A. 风格和谐度（40%）：512维向量距离
        style_dist = cosine_distance(existing_vector, cand.style_vector)
        score += (1 - style_dist) * 0.4
        
        # B. 色彩和谐度（30%）：HSV 欧氏距离
        color_dist = hsv_distance(existing_hsv, cand.color_primary_hsv)
        score += (1 - color_dist) * 0.3
        
        # C. 材质呼应度（20%）：材质是否同类或互补
        material_score = calculate_material_echo(existing_material, cand.material_tags)
        score += material_score * 0.2
        
        # D. 空间配合度（10%）：尺寸比例是否协调
        spatial_score = calculate_spatial_match(existing, cand)
        score += spatial_score * 0.1
        
        scored.append((cand, score))
    
    # 4. 排序取 Top K
    scored.sort(key=lambda x: x[1], reverse=True)
    return scored[:top_k]
```

### 材质呼应规则（预设）

示例代码

```text
def calculate_material_echo(material_a, material_b):
    """
    材质搭配得分：同类呼应 or 质感对比
    """
    # 同类呼应（+1.0）：柚木椅 → 柚木茶几
    if set(material_a) & set(material_b):
        return 1.0
    
    # 质感互补（+0.8）：实木椅 → 玻璃茶几（通透平衡厚重）
    complementary_pairs = [
        ({"实木", "柚木", "胡桃"}, {"玻璃", "亚克力"}),
        ({"藤编"}, {"金属", "不锈钢"}),
        ({"皮革"}, {"石材", "岩板"}),
        ({"布艺", "棉麻"}, {"实木"})
    ]
    for set_a, set_b in complementary_pairs:
        if set(material_a) & set_a and set(material_b) & set_b:
            return 0.8
    
    # 冲突预警（+0.2）：实木椅 → 塑胶茶几（质感落差大）
    return 0.2
```

### 空间配合规则（预设）

示例代码

```text
def calculate_spatial_match(existing, candidate):
    """
    尺寸比例协调度
    """
    if candidate.category == "茶几":
        # 茶几宽度 = 沙发宽度的 0.4-0.6 最佳
        sofa_width = existing.product_dimensions["w"]
        table_width = candidate.product_dimensions["w"]
        ratio = table_width / sofa_width
        if 0.4 <= ratio <= 0.6:
            return 1.0
        elif 0.3 <= ratio < 0.4 or 0.6 < ratio <= 0.8:
            return 0.6
        else:
            return 0.2
    
    elif candidate.category == "地毯":
        # 地毯应比沙发宽 20-40cm 每边
        pass
    
    return 0.5  # 默认
```

### 短边（0.9m）端头人数

表格

## 一、产品匹配算法（选品阶段）

**目标**：从 10 万张产品库中，为客厅选出风格和谐、预算匹配的候选款。

表格



| 步骤           | 算法动作            | 客厅案例                                                     |
| :------------- | :------------------ | :----------------------------------------------------------- |
| **元数据筛选** | SQL 硬条件过滤      | 客户要"中古风客厅"，预算 2-5 万 → 只保留 `style_label=中古风` + `scene_tags 含客厅` + `price_band=mid/high` 的产品 |
|                |                     | 沙发候选：200 款 → 15 款                                     |
|                |                     | 茶几候选：150 款 → 12 款                                     |
|                |                     | 地毯候选：80 款 → 8 款                                       |
|                |                     | 落地灯候选：60 款 → 6 款                                     |
| **向量精排**   | 512维风格向量算距离 | 以选定沙发（昌迪加尔椅风格皮沙发）为锚点，计算茶几/地毯/落地灯的风格向量距离 |
|                |                     | 茶几 Top 3：柚木藤编茶几（距离 0.08）、纯实木茶几（0.15）、玻璃金属茶几（0.28） |
|                |                     | 地毯 Top 3：波斯花纹羊毛毯（0.12）、几何棉麻毯（0.18）、纯色短绒毯（0.22） |
|                |                     | 落地灯 Top 3：实木落地灯（0.10）、金属工业灯（0.20）、布艺灯罩落地灯（0.25） |
| **推荐待选**   | 业务规则加权排序    | 材质呼应规则：沙发是皮革+实木 → 优先推柚木藤编茶几（同类呼应），淘汰玻璃茶几（质感冲突） |
|                |                     | 色彩距离：沙发 HSV=[30,0.6,0.4]（焦糖棕）→ 地毯主色 HSV=[25,0.5,0.35]（驼色）距离 0.08，和谐 |
|                |                     | 输出：每品类 3 款候选，共 12 款进入下一阶段                  |

**此阶段不计算尺寸、不检查碰撞，只看"好不好看、搭不搭调"。**

## 二、空间约束算法（环境适配阶段）

**目标**：检查每款产品与客厅硬装环境的耦合关系，提前预警。

表格



| 耦合类型         | 算法动作                 | 客厅案例                                                     |
| :--------------- | :----------------------- | :----------------------------------------------------------- |
| **与地面耦合**   | 材质适配 + 承重校验      | 客厅地面：效果图识别为 **哑光木地板**                        |
|                  |                          | 沙发：全落地框体，木地板承重无问题 ✅                         |
|                  |                          | 茶几：金属细腿 → 算法提示"金属腿+哑光木地板=易刮花，建议配毛毡垫" 🟡 |
|                  |                          | 落地灯：圆盘底座 35cm，重量 8kg → 压强 8kg/0.1m²=80kg/m²，木地板无压痕 ✅ |
|                  |                          | 地毯：直接铺放，无承重问题 ✅                                 |
| **与墙面耦合**   | 靠背/灯高 vs 插座/护墙板 | 沙发高背 95cm，墙面插座高度 75cm → **冲突检测**：靠背遮挡插座，使用不便 🔴 |
|                  |                          | 算法建议：改选低背沙发（<80cm）或调整电位                    |
|                  |                          | 落地灯高度 160cm，墙面无遮挡物 ✅                             |
|                  |                          | 茶几不靠墙，无墙面耦合 ✅                                     |
| **与顶面耦合**   | 层高 vs 产品高度         | 客厅层高 2.7m                                                |
|                  |                          | 落地灯 160cm + 沙发高背 95cm → 视觉净空 2.7-1.6=1.1m，无压迫感 ✅ |
|                  |                          | 如选吊灯（计划中）：吊灯垂高 60cm → 灯下净空 2.1m，需确认不碰头 ✅ |
| **水电点位耦合** | 插电产品 1m 半径检索     | 沙发：电动功能位需插电 → 算法标记"摆放位置 1m 内必须有插座"  |
|                  |                          | 落地灯：需插电 → 同样标记                                    |
|                  |                          | 茶几/地毯：无插电需求 → 无标记                               |
|                  |                          | **输出**：《现场确认清单》——电工需确认沙发位和灯位是否有插座 |

## 三、物理占位算法（布局校验阶段）

**目标**：计算产品放进客厅后，实际占用多少空间，产品之间是否打架。

### 3.1 静态占位（产品本身尺寸）

表格



| 产品       | 静态占位（宽×深×高）   | 算法用途         |
| :--------- | :--------------------- | :--------------- |
| **沙发**   | 2200×950×850mm         | 计算客厅能否放下 |
| **茶几**   | 1200×600×400mm         | 计算与沙发间距   |
| **地毯**   | 2000×3000×15mm         | 计算覆盖区域     |
| **落地灯** | Φ350×1600mm（直径×高） | 计算底座占地圆   |

### 3.2 动态占位（使用时的实际空间）

表格



| 产品       | 动态占位                                    | 算法用途           |
| :--------- | :------------------------------------------ | :----------------- |
| **沙发**   | 2200×1400×850mm（座深+腿部前伸 450mm）      | 计算电视观看距离   |
| **茶几**   | 1200×900×400mm（拉出抽屉/翘脚 +300mm）      | 计算通道是否被堵   |
| **地毯**   | 2000×3000mm（不变，但边缘需上卷 50mm 防绊） | 计算与茶几重叠区域 |
| **落地灯** | Φ600×1600mm（灯罩摆动/换灯泡半径）          | 计算与沙发扶手距离 |

### 3.3 产品间配合（正向 + 冲突 + 通道）

#### A. 正向配合（主动推荐）

表格



| 配合对            | 算法规则                                          | 计算结果                                                     |
| :---------------- | :------------------------------------------------ | :----------------------------------------------------------- |
| **沙发 ↔ 茶几**   | 茶几宽度 = 沙发宽度的 0.4-0.6                     | 沙发 2200mm × 0.55 = **1210mm** → 推荐 1200mm 茶几 ✅         |
| **沙发 ↔ 地毯**   | 地毯宽度 = 沙发宽度 + 每侧 200-400mm              | 2200 + 400×2 = **3000mm** → 推荐 2000×3000mm 地毯 ✅          |
| **沙发 ↔ 落地灯** | 灯罩高度 ≥ 沙发靠背高度 + 200mm（阅读照明不刺眼） | 850 + 200 = 1050mm → 灯罩应在 1050mm 以上 ✅（落地灯总高 1600mm，灯罩约 1200mm，符合） |

#### B. 冲突检测（硬性拦截）

表格



| 冲突对             | 算法规则                                              | 计算结果                                                     |
| :----------------- | :---------------------------------------------------- | :----------------------------------------------------------- |
| **茶几 vs 沙发**   | 茶几前沿距沙发座前沿 ≥ 300mm（起身+过人）             | 沙发座深 600mm，茶几深 600mm，客厅净深 3500mm → 剩余 3500-950-600-300 = **1650mm 通道** ✅ |
| **落地灯 vs 沙发** | 灯底座圆心距沙发扶手 ≥ 150mm（避免碰腿）              | 沙发扶手宽 80mm，灯底座半径 175mm → 中心距需 ≥ 255mm ✅       |
| **地毯 vs 茶几**   | 茶几必须压在地毯上（防滑+美观），压边 ≥ 100mm         | 地毯 2000×3000，茶几 1200×600 → 茶几放地毯中央，四周压边 400-900mm ✅ |
| **沙发 vs 电视墙** | 沙发动态占位后沿距电视墙 ≥ 2500mm（55寸电视最佳视距） | 沙发动态深 1400mm，客厅开间 4500mm → 视距 4500-1400 = **3100mm** ✅ |

#### C. 通道预留（动线校验）

表格



| 通道位置          | 算法规则                          | 计算结果                                                     |
| :---------------- | :-------------------------------- | :----------------------------------------------------------- |
| **沙发前主通道**  | 茶几动态占位后沿 → 电视墙 ≥ 800mm | 见上文，1650mm ✅                                             |
| **沙发侧通道**    | 沙发扶手 → 侧墙 ≥ 600mm           | 客厅宽 4500mm，沙发宽 2200mm → 两侧各 (4500-2200)/2 = **1150mm** ✅ |
| **入口 → 沙发区** | 入户门 → 沙发动态占位 ≥ 1000mm    | 入户门距沙发 1800mm ✅                                        |

### 手册核心结构

**一、双层录入架构**

- **RSPU（款式主档案）**：一次建档终身复用，不含任何工厂/价格信息
- **RSKU（工厂报价子表）**：每来一个工厂报价就新增一行，外键关联 RSPU

**二、RSPU 录入清单（4 层 20+ 字段）**

表格



| 层级   | 关键字段                                           | 谁录入                           |
| :----- | :------------------------------------------------- | :------------------------------- |
| 身份层 | `rspu_id`、`category_code`、`style_label`          | Agent 自动生成 + 人工确认        |
| 美学层 | `six_dim_tags`、`style_vector`(512维)、`color_hsv` | Agent 生成，人工微调             |
| 空间层 | `product_dimensions`、`occupancy_dynamic`          | 员工抄尺寸，Agent 自动算动态占位 |
| 业务层 | `budget_range`、`tier_badge`、`warranty_years`     | Agent 反推 + 工厂提供            |

**三、品类专属差异（12 个品类六维标签对照）**

- 休闲椅：6 维（轮廓/靠背/扶手/腿/材质/软包）
- 沙发：8 维（+座深 +模块性）
- 茶几：7 维（+高度 +配合比例）
- 吧椅：6 维（底座类型替代腿部，+座高档）
- 办公桌：6 维（隐私挡板 + 储物集成）
- 灯具/地毯/窗帘/装饰画：各 6 维，完全重定义

**四、RSKU 录入清单（供应商差异层）** 每厂一行：`rsku_id`（自动生成）、`factory_code`、`material_code`、`factory_price`、`structure_strength_rating`、`flame_retardant_capability`、**差异备注**

**五、个别差异标注规范**

表格



| 差异类型 | 标注方式                           | 示例                 |
| :------- | :--------------------------------- | :------------------- |
| 材质差异 | `material_code` 区分               | `TN`真藤 vs `PE`仿藤 |
| 结构差异 | `structure_strength_rating` + 备注 | 家用细框 vs 商用加厚 |
| 功能差异 | 差异备注字段                       | 旋转底座 vs 固定底座 |
| 尺寸差异 | ≤15% 同一 RSPU，>15% 新建 RSPU     | 标准版 vs 缩小版     |
| 颜色差异 | 备注"可定制5色"                    | 不改结构只改漆       |

**六、员工 5 步录入 SOP**

1. 上传白底图 → Agent 自动输出风格/六维/向量
2. 人工确认/修正
3. 同款判定（向量距离 <0.15 关联现有，≥0.15 新建）
4. 新建 RSPU（仅新款）
5. 录入 RSKU（每厂一行，填差异备注）

**七、批量优化策略**

- 同一工厂多款：工厂代码锁定，连续录入
- 同款不同厂：跳过 RSPU，只录 RSKU
- 工厂 Excel 报价单：Java 批量导入服务解析，员工只复核
