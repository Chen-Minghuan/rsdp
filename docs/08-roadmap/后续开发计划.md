## 一、P0 — 工厂角色与产品库去重（已确认需求）

### 1.1 背景与目标

当前 `FACTORY_SALES` 角色被设计成了“业务员”，实际需求是**工厂管理员**：
- 能管理自己的产品（弱归属：在该 RSPU 下已有本工厂 RSKU）。
- 能单条手动录入新产品（不走 AI），且必须同时录入第一条 RSKU。
- 批量导入产品仅管理员可执行。
- 可通过开关查看「全产品库去重视图」：隐藏自己能做的品类/风格/材质，只显示自己做不了的产品，只读，不能给别人报价。

### 1.2 数据模型变更

#### 1.2.1 `sys_user` 表

```sql
ALTER TABLE sys_user ADD COLUMN view_full_catalog BOOLEAN NOT NULL DEFAULT false;
```

#### 1.2.2 新增 `factory_product_capability` 表

```sql
CREATE TABLE factory_product_capability (
    id BIGSERIAL PRIMARY KEY,
    factory_code VARCHAR(16) NOT NULL,
    category_code VARCHAR(16),      -- 品类，如 OF
    style_code VARCHAR(16),         -- 风格/款式，如 EX
    material_code VARCHAR(8),       -- 材质，如 LE
    created_at TIMESTAMP DEFAULT NOW(),
    UNIQUE (factory_code, category_code, style_code, material_code)
);
```

能力行的粒度决定去重范围：
- 只填 `category` → 工厂能做整个品类。
- 填 `category + style` → 能做该品类下的该风格。
- 填 `category + style + material` → 仅能做该材质版本。

#### 1.2.3 角色/权限种子数据调整

不新增角色，将现有 `FACTORY_SALES` 重命名为 `FACTORY_ADMIN`，并调整权限：

| 角色 | 产品权限 | RSKU 权限 | 全库开关 | 能力表 |
|:---|:---|:---|:---:|:---:|
| `ADMIN` | 全部 + `product:import` | 全部 | ✅ | 可查看 |
| `FACTORY_ADMIN` | `product:read`、`product:create`、`product:update`（仅自己产品） | `rsku:*` | ✅ 自己控制 | 不可见 |
| `EDITOR` / `VIEWER` / `USER` | 保持现有 | 保持现有 | 保持现有 | 保持现有 |

### 1.3 后端设计

#### 1.3.1 能力同步

新增 `FactoryCapabilityService`：
- 自动同步：根据工厂已有 RSKU 对应的 RSPU，提取 `(category, style, material)` 组合，写入/更新 `factory_product_capability`。
- 触发时机：
  - RSKU 创建、更新、删除后异步/同步刷新。
  - 提供 `POST /api/v1/admin/factories/{code}/capabilities/sync` 手动触发。
- 管理员端只读展示能力表。

#### 1.3.2 产品列表接口增强

```
GET /api/v1/products?viewMode=own|full&factoryCode=xxx
```

- `viewMode=own`：返回“自己的产品”，可再用 `factoryCode` 筛选到具体关联工厂。
- `viewMode=full`：返回自己的产品 + 全库中未被能力覆盖的产品。
- 权限校验：
  - 工厂角色调用 `full` 时，必须 `sys_user.view_full_catalog=true`。
  - `factoryCode` 必须是当前用户关联的工厂之一。

去重逻辑：
- 计算当前用户关联工厂的能力键集合。
- 遍历全库 RSPU，若其 `(category, style, material)` 命中任一能力键，则隐藏。
- 保留未被覆盖的 RSPU，并始终保留自己的产品。

#### 1.3.3 工厂单条录入接口

```
POST /api/v1/products/factory-entry
```

权限：`product:create`（`FACTORY_ADMIN`）。

请求体包含：
- RSPU 基础信息：category、style、size、color、material、dimensions、productLevel 等。
- 产品图片（multipart，可选）。
- 第一条 RSKU：factoryCode、factorySku、factoryPrice、materialCode、materialDescription 等。

实现：在一个事务中创建 `rspu_master`、`rspu_variant`、`rsku_supply`、`image_assets`。

#### 1.3.4 编辑产品接口增强

```
PUT /api/v1/products/{rspuId}
```

后端增加校验：当前用户关联的工厂必须在该 RSPU 下存在 RSKU，否则 403。

### 1.4 前端设计

#### 1.4.1 账号设置页

- 开关：**“显示全产品库（去重）”**，绑定 `sys_user.view_full_catalog`。

#### 1.4.2 产品库页面

- 开关开启后切换为 `viewMode=full`。
- 增加下拉筛选：**我的全部工厂 / 工厂 A / 工厂 B**（仅影响 own 视图）。
- 全库视图下的产品：
  - 隐藏“编辑”按钮。
  - 隐藏“录入报价”按钮。
  - 仅支持查看详情。

#### 1.4.3 工厂单条录入页

- 分两步：产品信息 → 第一条 RSKU 信息。
- 必填字段校验，字典值自动校验。

#### 1.4.4 管理员端能力表（只读）

### A.3 数据范围与产品可见性

- **ADMIN** → `DataScope.ALL`
- **DESIGNER** → `DataScope.ALL`（只读访问产品和 RSKU，写权限受功能矩阵限制）
- **产品可见规则**：DESIGNER 与 ADMIN 在产品库中均只展示 `rspu_master.review_status = 'APPROVED'` 的产品；PENDING / REJECTED / DRAFT 仅 ADMIN 可在审核后台看到。
- 任何非 ADMIN/DESIGNER/FACTORY 角色 → 仍按当前逻辑 `PUBLIC_ONLY`（EDITOR/VIEWER 后续需单独调整）。

### A.4 关键交互流程

#### 流程 1：ADMIN 准备产品库与策略
1. ADMIN 登录 → 进入「产品库」→「批量导入」上传 Excel。
2. ADMIN 对产品进行审核，将合格产品标记为 `APPROVED`。
3. ADMIN 进入「产品集」创建主流搭配集合（如「轻奢主卧套餐」）。
4. ADMIN 进入「推荐打分配置」设定全局/品类权重（品类热度、工厂评分、利润率、库存、交期、风格匹配、是否新品、是否主推）。
5. ADMIN 在「系统管理」执行向量回填，为后续以图搜图做准备。

#### 流程 2：DESIGNER 创建搭配方案（AI 推荐路径）
1. DESIGNER 进入「方案」→ 创建新方案，填写客户/空间/预算等上下文。
2. 点击「AI 推荐选品」，输入需求或上传参考图，调用 agent 推荐接口。
3. 返回的候选产品进入「候选清单」，显示推荐分与推荐理由。
4. DESIGNER 浏览候选，点击「确认加入」或「移除」；确认后成为正式方案项。
5. 在方案中调整数量、选择 RSKU，保存方案。

#### 流程 3：DESIGNER 创建搭配方案（手动选品路径）
1. DESIGNER 进入「产品集」，查看 ADMIN 预置的主流搭配。
2. 或直接进入「产品库」，使用分类树、风格、材质、预算等筛选器查找产品。
3. 将产品加入方案，调整数量与 RSKU，保存方案。

#### 流程 4：从方案生成报价单
1. DESIGNER 在方案详情页点击「生成报价单」。
2. 系统根据方案项、数量、所选 RSKU 计算总价。
3. DESIGNER 可导出 Excel 报价单。

#### 流程 5：ADMIN 介入调整
1. ADMIN 在「搭配方案」列表看到所有方案。
2. 对设计师提交的方案可编辑、补充备注或删除。
3. 对产品数据错误可直接在「产品库」中修正。

## 附录 B：产品集（Collection）设计

> 目标：解决产品数量暴涨后管理员难以管理、设计师难以快速找到主流搭配的问题。

### B.1 数据模型

```sql
CREATE TABLE product_collection (
    collection_id       UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    collection_code     VARCHAR(32) UNIQUE,
    name                VARCHAR(100) NOT NULL,
    description         TEXT,
    category_codes      JSONB,          -- 关联品类，如 ["FS", "OF"]
    style_codes         JSONB,          -- 关联风格，如 ["MC", "IT"]
    target_segments     JSONB,          -- 面向客户群体，如 ["family", "company"]
    is_featured         BOOLEAN DEFAULT false,
    sort_order          INT DEFAULT 0,
    status              VARCHAR(20) DEFAULT 'ACTIVE',
    created_by          UUID REFERENCES sys_user(user_id),
    created_at          TIMESTAMPTZ DEFAULT now(),
    updated_at          TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE product_collection_item (
    id              BIGSERIAL PRIMARY KEY,
    collection_id   UUID NOT NULL REFERENCES product_collection(collection_id) ON DELETE CASCADE,
    rspu_id         VARCHAR(32) NOT NULL,
    sort_order      INT DEFAULT 0,
    created_at      TIMESTAMPTZ DEFAULT now(),
    UNIQUE (collection_id, rspu_id)
);
```

### B.2 权限

- ADMIN：`product_collection:write`（增删改查）
- DESIGNER：`product_collection:read`（列表、详情）

### B.3 主要接口

```
GET    /api/v1/product-collections              # 列表（支持 category/style/segment 筛选）
GET    /api/v1/product-collections/{id}         # 详情（含 item 列表）
POST   /api/v1/admin/product-collections        # 创建（ADMIN）
PUT    /api/v1/admin/product-collections/{id}   # 更新（ADMIN）
DELETE /api/v1/admin/product-collections/{id}   # 删除（ADMIN）
```

### B.4 前端页面

- **ADMIN**：`ProductCollectionManageView.vue` — 集合列表、创建/编辑弹窗、拖拽排序产品。
- **DESIGNER**：`ProductCollectionView.vue` — 卡片式浏览集合，点击进入集合详情，可一键将集合内产品加入方案。

## 附录 C：推荐打分配置设计

> 目标：让管理员可配置 AI/系统推荐产品时的综合评分维度与权重，未来可给每个维度加批注。

### C.1 数据模型

```sql
CREATE TABLE recommendation_score_config (
    config_id       UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    config_name     VARCHAR(100) NOT NULL,
    category_code   VARCHAR(16),        -- null 表示全局默认
    client_segment  VARCHAR(20),        -- null 表示所有客户群体（未来扩展）
    weights         JSONB NOT NULL,     -- 见下方示例
    is_default      BOOLEAN DEFAULT false,
    description     TEXT,
    created_by      UUID REFERENCES sys_user(user_id),
    created_at      TIMESTAMPTZ DEFAULT now(),
    updated_at      TIMESTAMPTZ DEFAULT now(),
    UNIQUE (category_code, client_segment)
);
```

### C.2 权重 JSON 示例

```json
{
  "category_heat_weight": 15,
  "factory_rating_weight": 15,
  "profit_margin_weight": 10,
  "inventory_weight": 10,
  "lead_time_weight": 10,
  "style_match_weight": 20,
  "is_new_weight": 10,
  "is_featured_weight": 10
}
```

约束：所有权重之和必须等于 100，由后端校验。

### C.3 权限

- ADMIN：`recommendation_config:write`
- 其他角色：只读或不暴露（agent 服务可通过内部接口读取当前默认配置）

### C.4 主要接口

```
GET    /api/v1/admin/recommendation-score-configs               # 列表
GET    /api/v1/admin/recommendation-score-configs/{id}          # 详情
POST   /api/v1/admin/recommendation-score-configs               # 创建
PUT    /api/v1/admin/recommendation-score-configs/{id}          # 更新
DELETE /api/v1/admin/recommendation-score-configs/{id}          # 删除
POST   /api/v1/admin/recommendation-score-configs/preview       # 预览某配置对指定产品的打分结果
GET    /api/v1/recommendation-score-configs/default             # 获取当前默认/适用配置（供 agent 调用）
```

### C.5 前端页面

- `RecommendationScoreConfigView.vue`（仅 ADMIN）：
  - 列表展示全局配置与品类覆盖配置。
  - 编辑页用滑动条/数字输入框调整 8 个维度权重，实时显示总和。
  - 提供「预览」功能：选择品类/风格/预算，查看产品排序与得分明细。

## 附录 E：AI 推荐候选清单接口（预留）

```sql
);
```

### E.2 主要接口（预留）
