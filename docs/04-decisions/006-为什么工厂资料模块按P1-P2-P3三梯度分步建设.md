## 背景

RSDP 现有工厂模块仅覆盖基础档案（17 字段：代码、名称、等级、联系人等），占项目总代码体量约 5%。用户提出的「工厂资料模块架构图」要求大幅拓展工厂信息维度，包括：

- **未验厂**：工厂标签信息（图片、核心产品、产地标签、产品类别、风格、材质、预算）
- **已验厂**（4 个子模块）：
  1. 工厂标签信息（含验厂记录）
  2. 工厂基本信息（规模/设备/原料/品控/认证/物流）
  3. 工厂交期规则（8 品类 × 6 维度交期表）
  4. 工厂面料材质组（4 色块分组 × 最多 16 材质码/组）

若一次性全部落地，预估需新增 3-4 张表、15-20 个 Java 文件、4-6 个前端页面，整体工作量约 13-31 个工作日（1-2 人），相当于扩展现有工厂模块 3-4 倍。

此外，模块之间存在依赖关系：AI 集成依赖基础数据完整，面料材质组校验依赖 RSKU 创建链路，交期规则依赖报价生成流程。一次性全量开发风险高、验证周期长，且 AI 模块是否必要仍需业务方后续确认。

## 决策

将工厂资料模块拆分为三个优先级梯度，**按 P1 → P2 → P3 顺序逐层建设**，每层可独立交付与验证：

### P1：工厂表信息补充 + 前端详情查看（近期优先）

- **工作范围**：仅扩展 `factory_master` 单表的字段（规模/设备/原料/品控/认证/物流/验厂），不新增子表
- **后端改动**：Entity/DTO/Service/Controller 各扩展约 50%，新增 1 个 `PUT` 更新接口
- **前端改动**：`FactoryDetailView.vue` 从当前简单卡片改为多信息卡片布局 + 编辑弹窗
- **不涉及**：交期规则、面料材质组、未验厂标签、AI 集成
- **预估工期**：1 人 3-5 天
- **理由**：
  - 纯 CRUD，无新表、无复杂校验、无并发问题，风险最低
  - 让工厂详情页从「只能看代码和名称」变为「能看到完整的工厂画像」，业务价值立即可见
  - 为 P2 交期规则和材质组提供数据基础（设备清单、原料来源等字段是后续关联校验的前提）

### P2：交期规则 + 面料材质组 + 验厂表单 + 未验厂标签（后续规划）

- **工作范围**：新建 3 张子表 + 扩展工厂验厂状态字段 + 7-8 个新 Service/Controller + 4-5 个新前端页面
- **涉及链路**：报价生成引用交期规则快照、RSKU 创建校验材质组范围、未验厂标签转正流程
- **并发处理**：工厂主表增加乐观锁 `@Version`、交期规则使用 MERGE 语义
- **预估工期**：1 人 10-15 天，2 人 7-10 天
- **理由**：
  - 业务逻辑复杂（跨表校验、状态流转、快照机制），需要 P1 基础数据完整后再开发
  - 并发控制方案需要在 P1 字段结构确定后统一设计
  - 可分批交付：先上交期规则（直接提升报价准确度），再上材质组（约束 RSKU 质量），最后验厂表单

### P3：AI 集成工厂上下文（低优先级，待业务决策）

- **工作范围**：prompt 工程、AI 匹配增加工厂适配度评分、检索支持工厂能力过滤
- **不涉及**：新表、新页面（仅修改现有 AI 服务类和前端搜索页）
- **预估工期**：1 人 3-5 天（不含调优）
- **理由**：
  - prompt 调优周期不确定，效果依赖基础数据质量
  - 当前 MVP 阶段「AI 看图识别产品 → 人工复核 → 匹配 RSKU」的链路已经跑通，工厂上下文是锦上添花
  - 放 P3 不影响 P1/P2 的独立交付能力

## 影响

- **P1 完成后**：工厂详情页成为真正可用的「工厂档案」，可独立上线供业务方使用，不依赖 P2/P3
- **P2 完成后**：报价准确度显著提升（交期不再依赖人工记忆），RSKU 质量受材质组约束
- **P3 待决策**：若业务方决定不做 AI 工厂上下文，P3 条目可直接关闭，不浪费开发资源
- 三个梯度的测试可独立编写与运行，不会因上层未完成而阻塞下层验收

## 待办

- P2 启动前确认面料材质组的 4 色块划分规则是否已有完整字典数据
- P3 启动前由业务方评估 AI 工厂上下文的投入产出比
