# RSDP 前端页面权限与业务目的复核报告

> 复核时间：2026-07-08  
> 复核范围：`web/src` 全部路由、导航、视图、API、常量与状态；对照后端 `SecurityConfig.java`、`Permissions.java`、`database/seed_data.sql`。  
> 复核结论：**权限框架基本成型，主要风险集中在工厂管理员数据范围隔离、前后端校验一致性、无权限提示三方面。**  
> 已确认方向：**不改权限矩阵，只加数据范围校验**（工厂管理员只能操作与本厂有关联的产品/工厂数据）。

---

## 一、当前权限模型（前后端一致）

| 角色 | 关键权限（来自 seed_data.sql） |
|------|-------------------------------|
| ADMIN | 全部权限 |
| EDITOR | 除用户管理、admin 后台、推荐打分配置外几乎全部权限 |
| FACTORY_ADMIN | `product:read/create/update`、`factory:read`、`rsku:*`、`capability:read` |
| DESIGNER | 读 + `quote:generate/export`、`scheme:*`、`designer:profile:*`、`scheme:candidate:*` |
| VIEWER / USER | 只读 |

- `web/src/utils/constants.ts` 中的 `ROLES`、`PERMISSIONS` 与后端字符串完全一致。
- `web/src/stores/user.ts` 提供 `hasRole / hasPermission / isPlatformStaff`。
- `web/src/router/index.ts` 通过 `meta.requiresAuth / roles / permissions` 做全局守卫。
- `web/src/App.vue` 顶部导航基本按权限/角色显隐。

---

## 二、高优先级问题（建议立即修复）

### 1. 工厂管理员缺少数据范围隔离 —— 可修改/删除不属于自己的产品

**表现**：`FACTORY_ADMIN` 拥有 `product:update/delete`，但目前后端未校验目标产品是否与本厂存在 `rsku_supply` 关联。工厂管理员在产品详情页可编辑/删除任何 RSPU，包括只由其他工厂供应的产品。

**业务影响**：同一 RSPU 是跨工厂共享的款式定义。若 A 工厂管理员修改或删除了 B 工厂唯一供应的产品，会直接影响 B 工厂业务。

**修复方向（已确认：不改权限矩阵，只加数据范围校验）**：
- **后端**：在所有 RSPU 写操作（编辑元数据、复核、删除、新增/编辑变体、添加/删除搭配关系）前校验：当前用户 `factoryCodes` 是否与该产品存在 `rsku_supply` 关联；平台运营人员（ADMIN/EDITOR）不受限制。
- **删除产品再加一道保险**：若该产品还关联了其他工厂的 RSKU，禁止删除并提示“其他工厂已关联/报价，请联系平台运营”；仅当只有本厂关联或本厂是创建者时才允许删除。
- **前端**：产品详情页各操作按钮在现有 `!isReadOnly` 基础上，增加“当前用户工厂是否关联该产品”的显隐/置灰判断；产品列表删除按钮同步处理。

**涉及文件**：`web/src/views/ProductDetailView.vue`、`web/src/views/ProductListView.vue`；后端 `ProductQueryService`、`RspuVariantService`、`RspuRelationService` 等。

---

### 2. 工厂/RSKU 数据范围隔离不足

**表现**：
- `ProductDetailView.vue` 新增 RSKU 时工厂下拉框未按当前用户 `factoryCodes` 过滤。
- `FactoryDetailView.vue` 编辑工厂、同步能力档案按钮未按 `factoryCodes` 判断是否归属。
- `RskuDetailView.vue` 更新/删除 RSKU 未校验是否属于当前用户工厂。
- `RskuImportView.vue` 批量导入时 factoryCode 任意。

**修复方向**：
- 后端所有涉及 `FACTORY_ADMIN` 的写接口必须校验目标 `factoryCode` 是否在当前用户 `factoryCodes` 内。
- 前端辅助隐藏不属于当前用户工厂的操作入口，并在工厂选择器中过滤为 `userInfo.factoryCodes`。

**涉及文件**：`web/src/views/ProductDetailView.vue`、`FactoryDetailView.vue`、`RskuDetailView.vue`、`RskuImportView.vue`；后端 `FactoryService`、`RskuService` 等。

---

### 3. `/factory-entry` 前后端校验不一致

**表现**：前端路由要求 `PRODUCT_CREATE + FACTORY_ADMIN`，后端 `POST /v1/products/factory-entry` 仅要求 `PRODUCT_CREATE`。

**修复方向**：推荐后端增加 `.hasRole("FACTORY_ADMIN")`，使工厂单条录入真正专属工厂管理员。

**涉及文件**：`server/src/main/java/com/rsdp/config/SecurityConfig.java`。

---

### 4. 用户管理按钮权限与后端实际权限不一致

**表现**：
- 前端「重置密码」按 `USER_RESET_PASSWORD` 显隐，但后端 `SecurityConfig` 的 `PUT /admin/users/**` 规则使其实际需要 `USER_UPDATE`。
- 前端「删除用户」按 `USER_DELETE` 显隐，但后端 `/api/v1/admin/**` 兜底需要 `ADMIN` 角色。

**修复方向**：调整后端 URL 规则或方法级注解，使实际权限与前端的 `PERMISSIONS` 语义一致。

**涉及文件**：`server/src/main/java/com/rsdp/config/SecurityConfig.java`、`server/src/main/java/com/rsdp/controller/UserController.java`、`web/src/views/UserManagementView.vue`。

---

## 三、中优先级问题

### 5. 缺少 403/404 页面，无权限时静默回首页

**表现**：`router.beforeEach` 权限/角色校验失败时直接 `next('/')`；非法路径也没有 404 页面。

**修复方向**：新增 `ForbiddenView.vue` / `NotFoundView.vue`，无权限时跳 `/403`，通配路由跳 `/404`。

**涉及文件**：`web/src/router/index.ts`、新增视图。

---

### 6. `/quotes/build?editSchemeId=xxx` 编辑模式进入门槛过低

**表现**：路由仅校验 `QUOTE_GENERATE`，但编辑模式需要 `SCHEME_READ` + `SCHEME_UPDATE`。

**修复方向**：路由守卫识别 `editSchemeId` 并校验 `SCHEME_UPDATE`；创建模式校验 `SCHEME_CREATE`。

**涉及文件**：`web/src/router/index.ts`、`web/src/views/QuoteBuilderView.vue`。

---

### 7. `RskuDetailView.vue` 未遵循「全库只读视图」

**表现**：工厂管理员开启 `viewFullCatalog` 后，产品库/产品详情为只读，但 RSKU 详情仍显示「删除报价」「更新价格」按钮。

**修复方向**：增加 `isReadOnly` 计算属性，操作按钮叠加 `!isReadOnly`。

**涉及文件**：`web/src/views/RskuDetailView.vue`。

## 五、整体重构建议

1. **短期（本轮）**：按“不改权限矩阵，只加数据范围校验”的方向，修复高优先级问题 1-4 + 403/404 页面。
2. **中期**：封装 `useCatalogViewMode()`、`useProductPermission()`、`useFactoryPermission()` 等组合式函数，统一数据范围与按钮权限判断。
3. **长期**：维护「页面-角色-权限-API」矩阵文档；增加 CI 脚本对比前后端权限字符串一致性。

---

## 六、是否重构？

**不建议推倒重写**。当前前端权限体系在路由、状态、常量、API 映射上已具备可维护基础，主要问题是：
- 部分写操作缺少按 `factoryCodes` 的数据范围校验；
- 缺少统一的数据范围与只读视图抽象；
- 无权限/错误状态的用户体验不足。

**推荐做法**：以“高优先级修复 + 权限组合式函数抽象”为最小改造单元，逐步优化，避免大范围重写带来的回归风险。
