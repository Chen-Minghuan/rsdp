> 审查方式：6 个只读审查代理并行覆盖【后端核心服务 / AI 与外部集成 / 安全与数据层 / 前端 / 数据库与基础设施 / 测试与工程质量】，以下为汇总结果。

## 修复状态（2026-07-23 更新）

**已修复（P0 全部 + P1 大部分）**：严重 #1（出厂价掩码）、#2（异步任务收割器+saveFailure）、#3（行级事务边界）、#5（回填游标分页）、#6（nginx 证书脚本）、#7（种子密码覆盖）、#8（密码进日志）、#9（Mock 模式）；另修复 RSPU 快照 MismatchedInputException、存储失败静默降级。
**严重 #4 复核修正**：`GenericJackson2JsonRedisSerializer` 无参构造实际已启用 NON_FINAL 类型信息（字节码 + 单测验证），原"全部 CCE"结论不成立；真实缺陷为 `PermissionService` 缓存 final 的 `Collections.emptySet()`，已改 `HashSet` 并加回归测试。
**待办（P2/P3）**：Security 401/403 语义统一、审计 DiscardPolicy 改 CallerRuns、异步线程 SecurityContext 传播、ChromaDB 检索降级、导入批次 importing 超时复位、前端轮询错误处理对齐/401 跳转/超时预算、CI workflow、Testcontainers 集成测试、核心 Controller 测试回填、备份恢复脚本与验证、docker-compose 健康检查与重启策略、确认导入异步化、prod 弱口令默认值。

## 总体结论

项目健壮性**整体中上，明显高于一般 MVP 水准**，防御性设计意识到位（原子抢占 SQL、事务外外部调用、超时+退避重试、SSRF/XSS 防线、密钥强制环境注入、幂等迁移脚本）。但存在若干**静默失效型**缺陷——平时不炸，一旦触发就是任务永久挂死、审计空洞、安全防线形同虚设。

**一句话：骨架结实，但"兜底链路"（异步恢复、审计必达、部署可用性）和"权限-加密一致性"是当前最薄的两块。**

## 🟡 中等问题（按主题归并）

**异步/任务生命周期**
- 导入批次可永久锁死 `importing`（JVM 崩溃场景，无复位接口、claim 不支持超时抢占）— `ExcelAiImportService.java:292-304`
- `updateTaskStatus` 读-改-写无并发保护，旧快照可覆盖终态 — `AsyncTaskProcessor.java:196-210`
- 确认导入在 HTTP 请求线程内同步跑完整个批次（数分钟，nginx 超时）— `ExcelAiImportController.java:59-63`

**安全**
- 审计日志线程池 `DiscardPolicy`，队列满时审计静默丢弃，违反"必须记审计"铁律（三处独立发现）— `AsyncConfig.java:84`
- 异步线程无 SecurityContext 传播，AI 回写审计操作人全记 `anonymous`；DataScopeContext ThreadLocal 在异步线程永不清除 — `AsyncTaskProcessor.java:66-67`
- Spring Security 未配 `authenticationEntryPoint/accessDeniedHandler`：token 过期返回 403 空 body，用户看到"权限不足"而非被引导重新登录 — `SecurityConfig.java`
- 生产 profile 数据库弱口令默认值 `${DB_PASSWORD:rsdp}` — `application-prod.yml:6`
- LoginAttemptService 内存 Map 无界增长（内存 DoS 面）；公开注册接口无限流 + 可枚举用户名
- Excel AI 导入 URL 图片缺魔数嗅探（标准通道有），SVG 存储型 XSS 缺口 — `ExcelAiImportService.java:2038-2068`
- RetryInterceptor 对非幂等 AI POST 的 read-timeout 重试 → 可能重复扣费 — `RetryInterceptor.java:42-48`

**功能健壮性**
- 以图搜图对 ChromaDB 零降级，向量库故障 = 检索整体 500 — `RetrievalService.java:76`
- StyleMatchingService.saveMatch 静默吞异常 + select-then-insert 竞态（无唯一约束）— `StyleMatchingService.java:419-447`
- saveImages 存储失败静默降级，用户看到"成功"但图片缺失 — `ExcelAiImportService.java:2642-2648`
- GlobalExceptionHandler 未处理 `DataIntegrityViolationException`（唯一约束冲突→"系统繁忙"）；除 ExternalServiceException 外全部返回 HTTP 200

**前端**
- 三处视图把"轮询请求失败"误标为"任务失败"并持久化到 localStorage（store 层已正确实现，视图层未对齐）— `ProductEntryView.vue:140-146` 等

- 备份脚本：custom 格式写 `.sql` 扩展名（误导恢复）、无轮转、无 restore 脚本、从未验证恢复

## ✅ 防御性设计亮点（值得保持）

1. **并发抢占全部用原子 SQL**（claimPendingTask/claimForImport），无 check-then-act
2. **AI/文件存储刻意排除在事务外**，DB 写入收敛为独立短事务（行级路径除外，见严重 #3）
3. 外部调用统一独立超时 + 指数退避重试（仅 5xx/429），RetryInterceptor 有测试
4. URL 图片下载完整防线：host 白名单、禁重定向、Content-Length 预检、限量读取、魔数嗅探（标准通道）
5. 越权统一返回 404 防存在性探测；邀请 token HMAC + 常量时间比较 + DB 只存哈希
6. JWT tokenVersion 机制，权限变更旧 token 立即失效；HttpOnly + SameSite Cookie
7. 密钥管理规范：JWT/AES/DashScope/MinIO 全部强制环境注入，启动期校验
8. 迁移脚本全面幂等，V1/reset/seed 三方同步执行到位，V17 预检+WARNING 是教科书式安全迁移
9. 前端 excelImport store：轮询代际令牌、pollError 与任务状态分离、超时后批次状态恢复
10. WireMock 落地（Vision/Embedding/ChromaDB），导入链路测试最厚（51+24 用例）
