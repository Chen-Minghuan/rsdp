# 双层编码体系 —— 整合分析报告

> 基于原始文档 `双层编码体系.md` 整合提炼，说明**要做什么**、**怎么落地**。

---

## 一、这个系统要做什么？

一句话：**为家具产品建立一套"款式 + 供应"双层数字化档案，让 AI Agent 能自动看图识别产品、判定同款、匹配搭配、校验空间。**

核心被拆成 **6 大目标**：

| # | 目标 | 解决的问题 |
|:--|:-----|:----------|
| 1 | **双层编码（RSPU + RSKU）** | 款式概念与工厂供应分离，一个款式对应多个工厂报价 |
| 2 | **AI 看图识别** | 新品入库时 Agent 自动输出风格、六维标签、512维向量 |
| 3 | **同款判定** | 防止同一款式被不同工厂重复建档 |
| 4 | **三层检索** | 从 10 万张产品库快速找到"最像的那款" |
| 5 | **空间方案生成** | 自动校验产品尺寸、动线、硬装冲突、搭配和谐度 |
| 6 | **批量化录入** | 员工按 SOP 流水线作业，Agent 承担大部分标注工作 |

## 二、核心概念：双层编码体系

### 2.1 编码结构

```
RSPU（设计原型）：[类别2位]-[风格2位]-[流水3位]-[尺寸1位]
  例：FS-MC-001-M  = 座椅-中古风-001号-中号

RSKU（供应单元）：[RSPU ID]-[工厂代码3位]-[材质版本2位]
  例：FS-MC-001-M-A004-PE = 同上款-A004厂-PE仿藤版
```

### 2.2 铁律

- **`rspu_master`** 绝对不含工厂、价格、交期、SKU
- **`rsku_supply`** 必须外键关联 `rspu_id`，携工厂 + 材质 + 价格

### 2.3 编码段详解

| 编码段 | 位 | 含义 | 示例 |
|:-------|:--|:-----|:-----|
| **类别** | 2位 | FS=座椅 / FT=桌几 / FC=柜类 / BS=吧椅 / OF=办公 / LT=灯具 / RF=软装 |
| **风格/定位** | 2位 | 家装：MC=中古 / BA=包豪斯 / IT=意式 / FR=法式 / WJ=侘寂 / NC=新中式 / CR=奶油 / IN=工业 / MP=孟菲斯 |
| | | 办公：EX=总裁级 / MG=经理级 / ST=职员级 / PU=公共区 / CO=会议 |
| **流水号** | 3位 | 001-999 |
| **尺寸档** | 1位 | S=小 / M=中 / L=大 / X=特大（吧椅用 6/7/8 表示座高） |
| **工厂代码** | 3位 | S001-S999（战略厂）/ A001-A999（核心厂）/ B001-B999（合作厂）/ C001-C999（备选厂） |
| **材质码** | 2位 | PE=PE仿藤 / LE=皮革 / TN=真藤 / NP=纳帕皮 / SU=磨砂皮 / MA=马鞍皮 / LI=亚麻 / SF=羊羔绒 / VE=天鹅绒 / WO=实木 / RK=藤编实木混血 / MT=金属 / PL=塑料 / CA=帆布 / AC=亚克力 |

## 三、数据库设计（SQLite 三张表）

### 3.1 `rspu_master`（款式主档案）
记录款式的**身份 + 美学 + 空间 + 业务**四层字段，不含工厂/价格信息。

关键字段：
- `rspu_id`（主键）、`category_path`、`style_label_human`、`six_dim_tags`
- `style_vector`（512维向量，Agent 自动生成）
- `color_primary_hsv`、`material_tags`、`scene_tags`
- `product_dimensions`、`occupancy_static`、`occupancy_dynamic`
- `hardscape_constraints`、`spatial_relationships`
- `reference_price_band`、`budget_range`、`warranty_years`

### 3.2 `rsku_supply`（工厂供应子表）
每行 = 一个工厂对一个 RSPU 的一种材质报价。

关键字段：
- `rsku_id`（主键）、`rspu_id`（外键）、`factory_code`、`material_code`
- `factory_price`、`lead_time_days`、`moq`
- `structure_strength_rating`、`flame_retardant_capability`
- `factory_credit_score`、`on_time_rate`、`quality_return_rate`

### 3.3 `factory_master`（工厂档案表）
独立维护工厂信息：等级、认证、区域、联系人、审核日期等。

## 四、多品类扩展：关键差异

不同品类不能套用同一套分类逻辑：

| 品类 | RSPU 第2段含义 | 六维标签特点 | 区别点 |
|:-----|:--------------|:------------|:------|
| **FS 休闲椅** | 风格（MC/CR/IT...） | 轮廓/靠背/扶手/腿/材质/软包（6维） | 基准模板 |
| **SF 沙发** | 风格 | +座深 +模块性（8维） | 同款硬指标 A+B+C+D+H |
| **TB 茶几** | 风格 | +配合比（7维） | 同款硬指标 A+B+C+D |
| **BS 吧椅** | 风格 | 底座替代腿部 + 座高档（6维） | RSKU 加第7段功能码（R旋转/L升降/F固定/S堆叠） |
| **OF 办公桌** | 职级（EX/MG/ST/PU/CO） | +隐私挡板 +储物集成（6维） | 不用风格用职级 |
| **办公家具** | 职级定位 | 材质码 3位 | SOL实木/VNR木皮/MEL三胺板/LEA头层真皮... |

**分类原则：**
- 家装家具 → 风格驱动（中古风/奶油风/意式）
- 办公家具 → 职级驱动（总裁级/经理级/职员级）
- 吧椅 → 座高 + 功能驱动（先锁座高匹配吧台，再按空间位置定功能）

## 五、AI Agent 如何工作

### 5.1 看图识别逻辑（不是测量，是模式匹配）

```
产品白底图 → 视觉编码器（像素→向量）→ 跨模态对齐 → LLM 输出标签
```

Agent 在预训练时见过数百万张家具图，靠 **"像素模式的统计相似性"** 认出产品特征。

### 5.2 六维标签判定（以休闲椅为例）

| 维度 | 判断方式 | 示例 |
|:-----|:---------|:-----|
| **A. 轮廓** | 看整体剪影 | A字架形 / 蛋形 / 方盒 |
| **B. 靠背** | 看背部高度与结构 | 编织镂空 / 高背包裹 / 无靠背 |
| **C. 扶手** | 看手臂支撑部位 | 平板直扶 / 无扶手 / 软包厚扶 |
| **D. 腿部** | 看支撑结构 | 八字外扩 / 四直腿 / 圆盘底座 |
| **E. 材质** | 看表面反光与纹理 | 藤编+实木 / 金属镜面 / 皮革光泽 |
| **F. 软包** | 看座面/靠背填充 | 无软包硬座 / 薄垫平板 / 全包裹 |

### 5.3 风格判定（9选1加权投票）

Agent 不是算分排序，而是多特征加权投票：
- 某风格得票显著高于其他 → 高置信度，直接输出
- 两风格票数接近 → 中置信度，标注"疑似边缘"
- 特征混杂无法归类 → 低置信度，标记"需人工确认"

### 5.4 局限性（必须知道）

| 场景 | Agent 表现 | 原因 |
|:-----|:----------|:-----|
| 工厂图光线差/角度刁钻 | 误判轮廓或材质 | 视觉编码器对低质量图敏感 |
| 新款设计杂交 | 风格投票分散 | 训练数据中没有这种组合 |
| 颜色严重偏色 | HSV 色彩提取失真 | 预训练以自然光图为基准 |
| 极小尺寸差异（腿粗差2cm） | 完全无法识别 | Agent 没有尺度标定能力 |
| 商用加厚版 vs 家用普通版 | 可能误判为不同款 | 结构差异在像素层面不明显 |

## 六、512 维向量本质（重要澄清）

**512维向量是模型的"黑箱指纹"，人类无法定义或修订每个维度代表什么。**

- 所有品类共享同一套 512 维空间
- 椅子落在"座椅山脉"区域，沙发落在"卧具平原"区域
- 区分品类不靠改维度，靠**元数据标签 + 向量距离联合筛选**
- 不同模型（CLIP vs BGE vs Qwen）的维度含义完全不同

**你能做的三件事（替代"修订维度"）：**
1. **提供锚点图片** → 建立品类/风格的语义质心
2. **定义品类专属的六维标签 + 检索权重** → 在可解释层控制
3. **元数据预筛** → SQL WHERE 硬条件先缩小范围

## 七、三层检索系统（核心算法链路）

### 7.1 三层漏斗

```
全库 100,000 张
    ↓ Layer 1: 元数据预筛（SQL WHERE）→ < 10ms
    ├─ 品类=休闲椅 → 剩 15,000
    ├─ 风格=中古风 → 剩 2,000
    ├─ 轮廓=A字架形 → 剩 200
    └─ 靠背=编织镂空 → 剩 12 张
    ↓ Layer 2: 向量精排（ChromaDB 512维距离）→ 30-80ms
    └─ Top 5 候选
    ↓ Layer 3: LLM 终审（Qwen 7B 多图比对）→ 5-10s
    └─ 输出判定：同款 / 疑似 / 新款
```

### 7.2 各层分工

| 层级 | 数据量 | 耗时 | 作用 |
|:-----|:------|:-----|:-----|
| 元数据预筛 | 10万→12 | < 10ms | **排错**：排除跨品类、跨风格 |
| 向量精排 | 12→5 | 30-80ms | **排近**：按像素相似度排序 |
| LLM 终审 | 5→1 | 5-10s | **排细**：语义级终审，输出理由 |

### 7.3 是否需要终审？

| 场景 | 需要终审？ | 原因 |
|:-----|:----------|:-----|
| **同款判定**（入库时） | ✅ 必须 | 防止重复建档 |
| **搭配推荐**（给客户时） | ❌ 不需要 | 需要开放性，近似款也应该出现 |

搭配推荐时 LLM 的角色从"判定同款"转为"生成搭配理由"。

## 八、三件套交付物

### 8.1 锚点图采集清单

### 8.3 三层检索代码框架
- `MetadataFilter` 类（SQLite 预筛）
- `VectorMatcher` 类（ChromaDB + CLIP 512维精排）
- `VisualJudge` 类（Qwen 2.5-VL 7B 多图终审）
- `RSDPRetrievalEngine` 主控编排器

## 十、空间智能三大算法

### 10.1 三大算法分工

| 算法 | 负责问题 | 输入 | 输出 |
|:-----|:---------|:-----|:-----|
| **产品匹配算法** | 选什么风格好看 | 风格+品类+预算 | 每品类3款候选 |
| **空间约束算法** | 硬装环境允不允许 | 效果图+产品属性 | 环境预警清单 |
| **物理占位算法** | 放进去打不打架 | 产品尺寸+空间尺寸 | 平面占位+冲突报告 |

### 10.2 空间约束内容

| 耦合类型 | 检查内容 | 示例 |
|:---------|:---------|:-----|
| **地面耦合** | 材质适配、承重 | 金属腿+木地板 → 建议配毛毡垫 |
| **墙面耦合** | 靠背 vs 插座高度 | 高背沙发95cm挡75cm插座 → 冲突 |
| **顶面耦合** | 层高压迫感 | 落地灯+高背组合的视觉净空 |
| **水电耦合** | 插电产品1m内需有插座 | 电动沙发/落地灯 → 输出现场确认清单 |
| **UV预警** | 材质日晒耐受等级 | 靠窗+纳帕皮 → 🔴高风险 |

### 10.3 物理占位内容

| 维度 | 内容 | 用途 |
|:-----|:-----|:-----|
| **静态占位** | 产品本身尺寸 | 计算能摆几件 |
| **动态占位** | 含使用操作空间 | 计算动线是否够 |
| **正向配合** | 茶几宽=沙发宽0.4-0.6 | 自动推荐搭配尺寸 |
| **冲突检测** | 餐椅座高差、推进深度等 | 硬性拦截不适配组合 |
| **通道预留** | 主通道≥800mm | 校验动线 |

### 10.4 客厅案例（沙发+茶几+地毯+落地灯）

- **产品匹配**：元数据筛选→向量精排→业务规则加权，输出每品类3款候选
- **空间约束**：识别木地板→茶几金属腿预警；沙发高背挡插座→冲突；落地灯+功能沙发→水电清单
- **物理占位**：茶几宽1210mm匹配沙发2200mm；地毯3000mm宽匹配沙发+400mm每侧；沙发前通道1650mm ✅

## 十一、餐桌椅耦合规则（重要）

| 耦合项 | 规则 | 冲突预警 |
|:-------|:-----|:---------|
| **座高差** | 餐椅座高 + 25~30cm = 餐桌高度 | <25cm → 卡腿；>30cm → 吃饭耸肩 |
| **推进深度** | 餐椅座深 ≤ 餐桌下方空档深度 | 椅子推不进去 |
| **位宽** | ≥ 60cm/人（舒适）/ 55cm（标准）/ 50cm（紧凑） | <50cm → 无法使用 |
| **椅背高度** | 靠背顶部 ≤ 桌面高度 + 15cm | 推进去后椅背顶桌面 |
| **扶手冲突** | 扶手宽度×2 < 桌下空档宽度 | 无法并排推进 |

**1.8m × 0.9m 餐桌容量：**
- 舒适档：6人（长边各3，端头放空） ← **默认推荐**
- 标准档：8人（端头可坐）
- 极限档：10人（不建议）

## 十二、员工批量化录入 SOP（5步）

| 步骤 | 动作 | 谁做 |
|:-----|:-----|:-----|
| **1. 上传白底图** | 拖入产品图 | 员工操作，Agent 自动输出风格/六维/向量 |
| **2. 人工确认** | 确认/修正 Agent 输出 | 员工 |
| **3. 同款判定** | 向量距离 <0.15 关联已有 RSPU，≥0.15 新建 | Agent + 员工确认 |
| **4. 新建 RSPU** | 填写尺寸/材质/场景等 | 员工 + Agent 辅助 |
| **5. 录入 RSKU** | 每厂一行，填工厂/价格/交期/差异备注 | 员工 |

### 个别差异标注规范

| 差异类型 | 处理方式 |
|:---------|:---------|
| 材质差异 | `material_code` 区分（TN真藤 vs PE仿藤） |
| 结构差异 | `structure_strength_rating` + 差异备注 |
| 功能差异 | 差异备注字段（旋转底座 vs 固定底座） |
| 尺寸差异 | ≤15% 同一 RSPU，>15% 新建 RSPU |
| 颜色差异 | 备注"可定制5色"，不改 RSPU |

## 十五、一句话总结

> 这套系统要做的就是把家具从"一张图和几个字"变成"AI 能读懂的数字档案"——RSPU 管款式概念（它长什么样），RSKU 管供应落地（谁做、多少钱、多久交），512维向量让机器"看懂"产品风格，三层检索让同款判定自动化，空间算法让方案不出错。最终目标是：**员工拍张照，Agent 自动建档、判重、选配、校验，人只做最终确认。**
